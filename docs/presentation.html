"회의에서 한 번에 얼라인 된 적 있어요?"

Problem

회의에서 사람마다 배경지식이 다르다

패턴 A: 설명 루프

한 명이 설명
→ "그게 뭔데?"
→ 다시 설명
→ 다른 사람은 이미 알아서 지루
→ 시간 다 써서 결론 못 냄

패턴 B: 침묵 통과

이해 못 한 사람이 그냥 넘어감
→ 나중에 엇나감
→ 재회의
→ 얼라인에 시간 2~3배

직장인 전원이 겪는 문제. 특히 PO · 개발자 · 디자이너 · 법무 등 크로스펑셔널 미팅.

Solution

팀 인텔리전스 레이어

회의 주제를 넣으면, 참석자 역할별 맞춤 브리핑 + 실시간 용어/맥락 설명이 나온다

Demo — 역할별 브리핑

같은 주제, 완전히 다른 브리핑

미팅 주제: "개인정보 마스킹 기능 도입"

🟣 PO

비즈니스 임팩트, 일정 영향, 우선순위 판단 기준, 경쟁사 대응 현황

🔵 BE 개발자

PII 마스킹 기술 구현, API 설계, 데이터 파이프라인, 성능 영향

🟠 디자이너

마스킹 UI 패턴, 사용자 인지 흐름, 접근성, 정보 표시 방식

⚪ 법무

개인정보보호법, GDPR 요건, 가명처리 vs 마스킹, 컴플라이언스

각 브리핑에는 핵심 요약 · 역할 관점 배경지식 · 예상 논의 포인트 · 용어 설명이 포함됩니다

Demo — 실시간 채팅 Q&A

같은 질문, 역할에 맞는 답변

질문

"PII 마스킹이 뭔데?"

BE 개발자

민감 데이터 필드(이름, 전화번호 등)를 해시 또는 토큰으로 치환하는 기술. 주로 DB 레벨 또는 API 응답 레벨에서 처리하며, 성능 오버헤드를 고려해야 합니다.

법무

개인정보보호법 제29조에 따른 안전성 확보 조치의 일환. 가명처리와는 법적 구분이 있으며, 마스킹은 원본 복원이 불가능한 비식별화 방식입니다.

디자이너

사용자에게 마스킹된 정보를 어떻게 표시할지의 UI 패턴. 예: •••• 또는 부분 노출(홍*동). 사용자가 "정보가 있다"는 것은 인지하되 내용은 보호.

Demo — 실시간 세션

스크립트 + 보조설명 + 궤도 이탈 방지

10:03:22 PO: "이번 스프린트에서 PII 마스킹 MVP를 넣으려고 합니다"

💡 BE개발자용 — MVP 범위에서 마스킹 대상 필드 정의가 선행되어야 합니다

💡 법무용 — 가명처리와 마스킹의 법적 구분을 확인하세요

마스킹 대상 필드 범위가 정해져 있나요? 이건 가명처리인가요, 마스킹인가요?
10:04:15 BE개발자: "DB 레벨에서 할지 API 레벨에서 할지..."

⚠️ 궤도 이탈 감지

구현 방식 논의가 시작되었으나, 아직 마스킹 대상 범위가 합의되지 않았습니다

→ "먼저 마스킹 대상 데이터 범위를 정하면 어떨까요?"
